# eval-runner

Golden-example evals for orchestration-GENERATED agents in a project.
Goldens are expected-PROPERTIES checklists, never exact transcripts —
generated agents are stochastic, so correctness is judged per property, not
by string match.

## Goldens

One file per golden at
`<target>/docs/orchestration/evals/<agent-name>/<golden-id>.json`:

```json
{
  "task": "<input task or dispatch brief>",
  "expectedProperties": ["<checkable property of a correct response>", "..."]
}
```

**Quota** — every generated agent (each blueprint specialist plus the
orchestrator) needs at least `evalRequirements.minGoldens` goldens; read the
number from that agent's entry in `docs/orchestration/blueprint.json`. Check
mechanically from the project root:

```
node --input-type=module -e '
import { readFileSync, readdirSync, existsSync } from "node:fs";
const bp = JSON.parse(readFileSync("docs/orchestration/blueprint.json", "utf8"));
let bad = false;
for (const a of [...bp.specialists, bp.orchestrator]) {
  const dir = `docs/orchestration/evals/${a.name}`;
  const n = existsSync(dir) ? readdirSync(dir).filter((f) => f.endsWith(".json")).length : 0;
  if (n < a.evalRequirements.minGoldens) { bad = true; console.log(`BELOW-QUOTA ${a.name}: ${n}/${a.evalRequirements.minGoldens} goldens`); }
  else console.log(`ok ${a.name}: ${n}/${a.evalRequirements.minGoldens}`);
}
const policy = bp.dispatch_rules?.routing_policy;
if (policy === "always" || policy === "threshold") {
  const dir = "docs/orchestration/evals/routing";
  const n = existsSync(dir) ? readdirSync(dir).filter((f) => f.endsWith(".json")).length : 0;
  if (n < 2) { bad = true; console.log(`BELOW-QUOTA routing: ${n}/2 goldens`); }
  else console.log(`ok routing: ${n}/2`);
}
process.exit(bad ? 1 : 0);
'
```

Flag every BELOW-QUOTA agent in the report. Quota gaps block the release
tier; the smoke tier still runs whatever goldens exist.

## Routing goldens (R-56)

A separate golden class at `<target>/docs/orchestration/evals/routing/` —
same `{task, expectedProperties}` shape, but the agent under test is the
**main loop**, not a generated agent: does it defer to the fleet when the
routing region says to? `"routing"` is a reserved agent name
(`validateBlueprint` rejects it), so this dir never collides with a real
agent's eval dir.

Required whenever `docs/orchestration/blueprint.json`'s
`dispatch_rules.routing_policy` is `always` or `threshold` — at least 2
goldens: one request that should qualify (defer to `tasks.md` +
`feature-orchestrator`) and one that should not (proceed inline). `manual`
policy emits no routing region, so routing goldens are exempt.

Run each golden from the scratch copy's root, same mutation-isolation rule
as above, but invoke the MAIN LOOP, not a named agent — no `--agent` flag, so
`AGENTS.md` (carrying the routing region) loads normally:

```
claude -p "<task from the golden>" --max-turns <small cap> --allowedTools <a conservative set — this must not actually execute the whole orchestration>
```

Judge from the transcript plus `git diff <base>`, same as agent goldens.
Affected-set rule for the smoke tier: routing goldens are affected whenever
`renderOrchestrationRouting` (`scripts/lib/orchestration/scaffold.mjs`), the
orchestrator template's `description:`, or the rendered AGENTS.md region
changes.

## Tiers

Both tiers are part of this procedure — pick by purpose, never skip the
tier the situation calls for:

1. **Smoke** — run each golden ONCE for the agents affected by an edit;
   every run must pass. This is the pre-merge regression check whenever a
   FleetCore template or a meta-skill changes. Affected agents: those whose agent
   template, paired skill template, or blueprint entry the edit touches;
   when a meta-skill changed, treat all generated agents as affected.
2. **Release** — run each golden 5 TIMES for every generated agent; a
   golden passes at a pass rate >= 4/5. A single pass is meaningless for
   gating a stochastic agent — it proves only that the agent can pass once,
   so never present a 1-run result as a release verdict.

## Run

**Mutation isolation — every golden runs in a disposable git copy of the
target, never in the live target.** Some goldens mutate state (the
orchestrator's end-to-end goldens commit, move tasks lines, append to the
handoff log), and any agent can misbehave — that misbehavior must land in a
throwaway copy, and each repeat of a golden must start from the same clean
base. Per run: `cp -R <target> <scratch>`, record the base commit, invoke in
the scratch copy, judge from its transcript plus `git diff <base>` and
untracked files, then delete the copy.

One invocation per run, from the scratch copy's root:

```
claude --agent <agent-name> -p "<task from the golden>" --allowedTools <agent's tools from its blueprint entry>
```

**Session lifetime — finish the batch before ending the turn.** When this
procedure itself runs headless, background jobs die with the session: never
end a turn with runs still pending in background shells. Either block until
every run has completed and judge in the same session, or hand the batch to
a driver that outlives the session and judge in a later invocation.

Then judge — the runner is the judge. For each property in the golden's
`expectedProperties`: one property = one verdict (HOLDS or FAILS, with one
line of transcript evidence). A run passes only if EVERY property holds.
Judge only what the property states; never grade against a remembered
"right answer" or transcript similarity.

## Report

One table per agent, plus one for `routing` when routing goldens ran:

| golden | runs | passes | verdict |
|---|---|---|---|
| <golden-id> | 1 or 5 | <n> | pass / FAIL |

Verdict rule: smoke — pass iff passes = runs; release — pass iff
passes >= 4 of 5. After the tables, list BELOW-QUOTA agents from the quota
check. Overall verdict is FAIL if any golden fails its tier's rule or (in
the release tier) any agent is below quota; include the failing properties
verbatim so the caller can fix the template or the golden.
